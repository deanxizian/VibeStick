# VibeStick v0.1.5 完整代码审查

审查日期：2026-07-17（Asia/Shanghai）

## 结论

本轮覆盖 Mac Bridge、Codex 状态观察、录音与语音转写、HTTP 协议、StickS3 固件、安装/诊断脚本、CI、文档和真机部署。发现的高风险与主要中风险问题均已修复；最终交叉复核没有遗留 P0/P1。项目目前适合作为个人可信局域网内的原型持续使用。

## 已完成的主要修复

| 范围 | 原问题 | 当前处理 |
| --- | --- | --- |
| 任务与提醒 | 轮询会重复解析大量 JSONL；子任务和过程状态可能误触发声音；多个提醒可能被覆盖 | 增加指纹缓存和有界缓存；只观察根任务终态；提醒改为有序队列并延长展示窗口；录音期间延迟播放 |
| 额度状态 | 旧快照或非有限数值会造成错误百分比 | 增加新鲜度、数值校验和失败回退；无有效 5 小时额度时显示 `--` |
| 录音会话 | 并发 start/stop、重启、响应丢失和设备复位可能锁死会话或重复粘贴 | 会话加锁并严格校验 `session_id`；持久化 `stopping`/粘贴保护；终态重试幂等；异常收敛为终态；Stick 端失败时保留 PCM 供重试 |
| 音频任务 | 固件停止时可能释放仍由录音任务使用的缓冲区 | 增加任务完成同步，停止与上传前等待任务退出；关键分配、队列和任务创建均检查失败 |
| 外部命令 | 本地转写/录音 hook 的输出无上限，超时可能遗留子进程 | 统一使用受限执行器；限制命令、JSON 输入和 stdout/stderr；独立进程组按 TERM→KILL 清理 |
| HTTP 与协议 | `/state` 可未授权读取；请求体/响应字段缺少边界；非 2xx 或截断响应可能被当作成功 | `/state` 令牌保护；严格 Content-Type、JSON、并发和体积限制；仅接受完整 2xx；设备字段按 UTF-8 字节限界 |
| 本地数据 | 状态/录音/配置权限与写入原子性不足，转写文本默认持久化 | 私有目录/文件使用 0700/0600 和原子替换；默认不保存 transcript；录音按保留期清理 |
| 安装与运维 | 安装可能误把占端口的旧进程视为成功；配置重复键有歧义；失败时缺少可靠回滚 | 事务式 staging/backup/rollback；安装 nonce、版本、PID 与受保护接口联合验真；拒绝重复配置；doctor 失败返回非零 |
| CI 与文档 | 运行时、固件和文档约束不完全一致 | CI 覆盖 Python 3.11–3.14、shell、Swift 和 ESP-IDF 5.5.1；协议、安全和操作文档同步到 0.1.5 |

## 接受的剩余风险

1. 提醒仍没有设备 ACK，因此不是严格的 exactly-once 投递。当前 2 秒轮询、6 秒展示窗口和有序队列能覆盖常见抖动，但 Bridge/设备重启或持续断网仍可能漏响；设备首次连接会建立基线并抑制已有旧提醒，以避免重启后补响。
2. Stick 与 Bridge 使用明文 HTTP。令牌可防止普通未授权调用，但不能抵抗同一不可信 Wi-Fi 上的嗅探和重放；应仅在可信局域网使用。
3. Codex 额度来自本地 session 事件；上游格式变化可能使额度暂时显示 `--`，但不会影响任务状态和录音。
4. 当前是单 factory 分区且仅余约 11.3% 空间，没有 OTA 回滚能力；继续增加固件功能前应调整分区或做体积预算。
5. 自动化测试没有覆盖真实云 ASR、目标应用的 Accessibility 粘贴以及所有按键/断网组合；本轮完成了真机启动和在线轮询验收，但这些仍属于硬件在环测试范围。
